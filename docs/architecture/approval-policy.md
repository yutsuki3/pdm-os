# 承認ポリシー

PdM OSの原則（[docs/vision/principles.md](../vision/principles.md) 原則2）に基づき、各承認ゲートは常に人間が最終判断する。本ドキュメントは、どのゲートで・誰が・何を基準に承認するかを定義する場所である。

【提案】承認ゲートを状態遷移上の5箇所（要件確定・仕様書承認・受領判断・QA依頼送付可否・リリース可否）に置く、というゲートの**構造**自体はClaude Codeによる設計提案であり、[state-machine.md](state-machine.md) の状態遷移と対応させている。この構造はユーザーの承認待ちであり、確定ではない。

【TBD】各ゲートの**承認者・承認基準**（誰が、何を基準に承認するか）は組織固有のルールであり不明なため、以下はすべてTBDとして記録する。ゲートの構造（提案）と承認者・基準（未決定）は別の確からしさのレベルにあることに注意する。

## 承認ゲート一覧

| ゲート | 対応する状態遷移 | 承認者 | 承認基準 | 備考 |
|---|---|---|---|---|
| 要件確定 | `requested` → `requirement_defined` | TBD | TBD | PdM単独か、関係者レビューが必要か不明 |
| 仕様書承認 | `spec_review` → `spec_approved` | TBD | TBD | PdMのみか、エンジニア/デザイナーの合意も必須か不明 |
| 受領判断 | `acceptance_review` → `accepted` / `rejected` | TBD (想定: PdM/PO) | TBD | 受領チェックリストの合格基準は [workflows/acceptance.md](../workflows/acceptance.md) 側もTBD |
| QA依頼の送付可否 | `accepted` → `qa_requested` | TBD | TBD | 受領後に必ずQAを挟むのか、条件によってスキップするのか不明 |
| リリース可否 | `qa_passed` → `release_note_drafting` → `released` | TBD | TBD | リリース判定会議の有無、承認フローは不明 |

## エージェントの権限境界

- いずれのエージェントも、上記ゲートの状態を**提案**することはできるが、状態を確定させる操作（承認・受領・リリース）は行わない。
- エージェントが生成する成果物（仕様書草稿、受領チェック結果、QA依頼文書、リリースノート草稿）は、常に「承認待ち」の状態で人間に提示される。

## 未確定事項（まとめ）

- 各ゲートの承認者（役職・個人）
- 承認に必要な人数・合議の要否
- 承認の記録方法（Notion上のプロパティか、Jiraのワークフローか、それ以外か）
- 緊急時（ホットフィックス等）に承認フローを簡略化する例外規定の有無
