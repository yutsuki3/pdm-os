# 正本 (Source of Truth) マッピング

## 原則

PdM OSはデータを複製・保有しない。各ドメインについて、正本となる外部システムを1つに定め、常にそちらを参照する。このリポジトリ内の草稿・テンプレート・生成物は中間成果物であり、正本ではない。

## マッピング表

【事実】ドメインと正本の対応はユーザーから明示された。

| ドメイン | 正本システム | 読み書き | 主な利用エージェント | 備考 |
|---|---|---|---|---|
| 現行仕様と意思決定 | Notion | 読み書き（仕様書は書込み対象） | Specification Agent, Knowledge Agent | 仕様書の正式版はNotion上に存在する前提 |
| 過去仕様 | Confluence | 読み取りのみ | Knowledge Agent | 現行仕様の代替にしない。経緯確認用 |
| 受領原本とGoogle Docs | Google Drive | 読み取り（受領判断対象） | Acceptance Agent | 「受領原本」の具体的なファイル種別・命名規則はTBD |
| タスク状態 (Jiraチケットのステータス) | Jira | 読み書き（タスク作成・状態参照） | Orchestrator, Specification Agent | プロジェクトキー・課題タイプ体系はTBD |
| 実装事実 | GitHub | 読み取りのみ | Acceptance Agent | PR/コミット/リリースタグのどれを「実装事実」の単位とするかはTBD |
| PdM OS自体の設計・ルール・状態管理 | このリポジトリ | 読み書き | 全エージェント共通 | Work Itemライフサイクルの**定義**はこのリポジトリが正本。実際の状態インスタンスの永続化方法は未確定 (下記参照) |

## 未割当の情報種別

【事実】上表には、QA結果、QA依頼の送付記録、要求原文、公開済みリリースノートの正本が割り当てられていない。したがって、これらをこのリポジトリ、Notion、Jira等のいずれかの正本として扱うことはできない。

| 情報種別 | 現在の正本 | 影響 | 対応するTBD |
|---|---|---|---|
| 要求・要望の原文 | TBD | 要件化前の出所を一意に辿れない場合がある | TBD-052 |
| QA依頼の送付記録 | TBD | `qa_requested` の完了証跡を一意に検証できない | TBD-051 |
| QA結果 | TBD | `qa_passed` / `qa_failed` の完了証跡を一意に検証できない | TBD-050 |
| 公開済みリリースノート | TBD | `released` の公開先・アーカイブを一意に辿れない | TBD-053 |

これらは正本の追加を提案・決定するものではない。正本が確認されるまで、成果物には外部参照または `TBD` を残す。

### PdM OS自体の正本について（補足）

【決定】Work Itemの状態遷移ルール・エージェント責務・承認ポリシーなど、PdM OS自体の設計は、外部システムではなくこのリポジトリを正本とする（ユーザーの指示に基づく）。

【仮定】これは「Work Itemライフサイクルの**定義**（どんな状態があり、どう遷移するか）」の正本がこのリポジトリである、という意味であり、実行時に生成される個々のWork Itemインスタンスの状態をどこに永続化するか（DB・Notion・Jira等）とは区別している。後者はAPI/DB設計を確定しない現フェーズでは `TBD`。
一方、個々のJiraチケットの進捗ステータスそのものは引き続きJiraが正本（上表の「タスク状態」行）。両者の関係（Work Item全体の状態と、配下の複数Jiraチケットの状態集約）は [docs/architecture/state-machine.md](state-machine.md) の未確定事項を参照。

## 各正本の役割詳細

### Notion（現行仕様と意思決定）

- 「今、何が正しい仕様か」を確認する唯一の場所。
- 仕様書のドラフトが承認された場合、最終的にNotion側へ反映される想定（反映方法・ページ構成・テンプレートはTBD）。
- 意思決定ログ（なぜその仕様にしたか）もNotion側に存在する前提だが、具体的な置き場所・フォーマットはTBD。

### Confluence（過去仕様）

- 現行仕様がNotionに移行する前、または過去のプロジェクトの仕様が保管されている想定。
- 検索対象ではあるが、現行仕様との優先順位は常にNotion > Confluenceとする。
- Confluenceのスペース構成・検索範囲はTBD。

### Google Drive（受領原本とGoogle Docs）

- 受領判断の対象となる成果物の原本（デザインファイル、Google Docsの提案書等）が置かれる場所。
- 「受領原本」に該当するファイルの見分け方（フォルダ構成・命名規則・ステータスプロパティ）はTBD。

### Jira（タスク状態）

- デザイン・実装タスクの状態管理はJiraが正本。
- 仕様書からタスクを作成する際のプロジェクト・課題タイプ・カスタムフィールドの体系はTBD ([templates/jira-task.md](../../templates/jira-task.md) 参照)。

### GitHub（実装事実）

- 「実際に何が実装されたか」はGitHubのコミット・PR・タグを正本とする。
- 受領判断の際、仕様書とGitHub上の実装差分を突き合わせる。対象リポジトリの範囲、PRのどの状態（マージ済み/レビュー済み等）を「実装事実」とするかはTBD。

## 正本間で情報が矛盾する場合・重複する場合

【決定】（[ADR-0003](../decisions/ADR-0003-conflict-logging-over-silent-resolution.md)）矛盾を検知した場合、黙って解決せず、必ず記録する。ドメインごとの正本を優先するが、優先順位を適用した事実自体も記録に残す。

【決定】複数のシステムから同じ情報が得られる場合（検索経路の重複）は許可する。ただし、同じ出所に由来する情報を複数の独立した根拠として扱わない。詳細な運用方針は [docs/architecture/deduplication-policy.md](deduplication-policy.md) を参照。

- 矛盾の記録を誰がレビューし、いつまでに解決するか（エスカレーション先・SLA）。
