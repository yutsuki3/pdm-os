# AGENTS.md

このファイルは、このリポジトリで作業するすべてのAIエージェント（Claude Code、その他のコーディングエージェント、将来実装される自動化エージェント）に共通する運用ルールを定義します。ツール固有の追加指示は [CLAUDE.md](CLAUDE.md) を参照してください。

## このリポジトリの性質

`pdm-os` は **設計リポジトリ** です。現時点でコードは実装していません。ドキュメント・スキーマ・テンプレート・プロンプトのみで構成されています。

- `src/` にコードを追加する作業は、明示的な依頼がない限り行わないでください。
- 「設計を進める」依頼は、原則として `docs/` `schemas/` `templates/` `prompts/` `examples/` の追加・更新を意味します。

## 最重要ルール: 不明な業務ルールは推測しない

PdM/POの業務には、組織固有の承認フロー・権限・SLA・ツール連携の詳細など、コードや会話から自明に導けない業務ルールが多く含まれます。

- 業務ルールが明記されていない、または文脈から一意に確定できない場合、**絶対に推測で埋めない**でください。
- 該当箇所には `TBD` と明記し、何が不明で、誰が決めるべきかが分かるように短い注記を添えてください。
  - 例: `承認者: TBD (プロダクト責任者か? PdMか? 要確認)`
- `TBD` を解消する際は、必ずユーザーとの会話で確認された内容に基づいて更新してください。

## 情報の正本 (Source of Truth) を尊重する

このシステムが扱うデータには、それぞれ正本となるシステムがあります。詳細は [docs/architecture/source-of-truth.md](docs/architecture/source-of-truth.md) を参照してください。

| ドメイン | 正本 |
|---|---|
| 現行仕様と意思決定 | Notion |
| 過去仕様 | Confluence |
| 受領原本とGoogle Docs | Google Drive |
| タスク状態 | Jira |
| 実装事実 | GitHub |
| PdM OS自体の設計・ルール・状態管理 | このリポジトリ |

- このリポジトリ内のドキュメント・テンプレート・生成物のうち、**PdM OS自体の設計・ルール・状態管理の定義を除き**、正本の写し・草稿・中間成果物です。正本そのものではありません。
- PdM OS自体（業務フロー・状態遷移・エージェント責務・承認ポリシー等の定義）については、このリポジトリが正本です。
- 仕様書・受領判断などを扱うドキュメントを書く際は、「どこが正本か」を常に明示してください。
- 複数のシステムから同じ情報が得られる場合があります。**検索経路が複数あることは問題ありませんが、同じ出所の情報を複数の独立した根拠として扱わないでください。** 情報の種類ごとに正本を優先し、矛盾は黙って解決せず記録してください。詳細は [docs/architecture/deduplication-policy.md](docs/architecture/deduplication-policy.md)。
- 正本間で情報が矛盾する場合、解決ルール自体は [ADR-0003](docs/decisions/ADR-0003-conflict-logging-over-silent-resolution.md) により「黙って解決せず記録する」と決定済みです。ただし記録先・エスカレーション先などの運用詳細は `TBD` です。推測で埋めないでください。

## 事実・仮定・提案・決定を区別する

業務ルールに関する記述は、次の4種類を明確に区別してください。文中で紛らわしい場合は `【事実】` `【仮定】` `【提案】` `【決定】` のラベルを付記します。

| ラベル | 意味 | 例 |
|---|---|---|
| 【事実】 | ユーザーが明示的に述べた、または既存資料から直接確認できる内容 | 【事実】正本はNotion（ユーザーが明示） |
| 【仮定】 | 確認が取れていないが、作業を進めるために暫定的に置いている前提。誤っている可能性がある | 【仮定】受領原本はGoogle Drive内の専用フォルダに置かれる |
| 【提案】 | Claude/Codex等が設計として提案している内容。ユーザーの承認を経ていない | 【提案】承認ゲートを5箇所に設ける |
| 【決定】 | ユーザーが承認した、またはADRとして確定した内容 | 【決定】矛盾は黙って解決せず記録する（ADR-0003） |
| `TBD` | 現時点で情報がなく、上記いずれにも分類できない未決定事項 | 承認者: `TBD` |

- 「決定」を勝手に作り出さないこと。ユーザーの承認・既存ADRの裏付けがない内容を「決定」と書かない。
- 未決定事項は [docs/decisions/tbd-register.md](docs/decisions/tbd-register.md) に集約する。新しいTBDを追加した場合はこの一覧も更新する。
- 重要な設計判断（正本の追加、矛盾解決の方針、エージェント責務の変更など）はADRとして `docs/decisions/` に記録する。

## ドキュメントを追加・編集するときのルール

- 既存のディレクトリ構成 (README記載の構造) に従い、新規ファイルは適切な場所に置いてください。構成を勝手に変更しない。
- Markdownの見出し構造は既存ファイルに合わせる。
- スキーマは `schemas/*.schema.yaml` に JSON Schema 相当の構造で記述する。
- テンプレートは `templates/*.md` に、実際にPdM/POが埋めて使える形で記述する。
- プロンプトは `prompts/*.md` に、対象エージェント・入力・出力・禁止事項を明記する形で記述する。
- ADR (Architecture Decision Record) を追加する場合は `docs/decisions/` に `ADR-XXXX-<slug>.md` の形式で追加し、`docs/decisions/index.md` を更新する。

## 人間の意思決定を代替しない

このシステムの目的は「支援・自動化」であり、PdM/POの意思決定を代替することではありません。

- 承認・受領判断・リリース可否などのゲートは、常に人間が最終決定する前提で設計する。
- エージェント定義 (`docs/agents/*.md`) やプロンプト (`prompts/*.md`) には、どこまでがエージェントの権限で、どこから人間の承認が必要かを明記する。

## モデルに依存しない設計

`docs/agents/*.md` で定義するのは **機能的な役割**（Orchestrator, Knowledge Agent, Specification Agent, Acceptance Agent, Release Agent）であり、特定のAIモデル・ツールへの依存を前提にしない。将来Claude・Codex・Gemini・Notion AI等の担当が変わっても、役割定義と業務フロー自体は維持できるようにする。現時点でのツールごとの暫定的な担当割り当ては [docs/agents/ai-tool-roles.md](docs/agents/ai-tool-roles.md) に別途記録し、機能的な役割定義と混同しない。

## この段階のスコープ

現在は設計フェーズの初期段階であり、以下は明示的に対象外とする（過剰な実装設計をしない）。

- API・MCP連携の具体的な実装方式
- 認証・権限管理の方式
- データベース・永続化層の設計
- クラウド構成・インフラ構成

これらはCodex以降のフェーズで、Schema・ファイル構造・文書間整合性のレビューと合わせて具体化される想定。このリポジトリでは、業務フローの構造化・状態遷移設計・成果物と責務の整理・承認ポイントの設計・不足している業務ルールの抽出・文書とテンプレートの初期案作成にとどめる。

## 用語

- **要求・要望**: ステークホルダーから寄せられる、まだ整理されていない発話・依頼。
- **要件**: 要求・要望を整理し、実現すべきこととして合意された内容。
- **仕様書**: 要件を実装可能な形に落とし込んだドキュメント (正本は Notion)。
- **ワイヤーフロー**: 画面遷移・操作フローを表現した資料。
- **受領判断**: 実装・デザイン成果物が仕様を満たしているかをPdM/POが判断すること。
- **QA依頼**: 受領判断後 (または受領判断のため) にQAチームへ検証を依頼すること。
